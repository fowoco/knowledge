# Intent·Domain 라벨링 가이드 V1

## 1. 라벨링 대상

대표 모델은 HR 원문과 화면 상황을 보고 하나 이상의 `Intent`와 `Domain`을 분류합니다.
문장에서 법적 결론을 내리거나 Workflow를 실행하는 것은 분류 모델의 역할이 아닙니다.

검수 순서는 다음과 같습니다.

1. 입력 모드 확인
2. 사용자가 원하는 행동인 Intent 선택
3. 그 행동이 다루는 업무 대상인 Domain 선택
4. 서로 독립적인 요청이 두 개 이상이면 복수 Intent 선택
5. 서비스가 수행하면 안 되는 최종판단·자동제출이면 `OUT_OF_SCOPE` 선택

## 2. Intent 결정표

| Intent | 선택 기준 | 선택하지 않는 경우 |
| --- | --- | --- |
| `WORKER_ONBOARDING` | 문서에서 근로자 등록 초안을 만들거나 기본정보를 변경 | 근로자에게 문서를 내라고 요청하는 경우 |
| `EXPIRY_RENEWAL` | 체류·계약 만료자를 찾거나 갱신 준비를 시작 | 퇴사·이탈·계약종료 사건의 후속조치 |
| `DOCUMENT_REQUEST` | 서류를 요청·확인·추적하거나 일반 증명서 발급을 준비 | 문서를 읽어 근로자 프로필을 등록 |
| `PAYROLL_EXPLANATION` | 기존 명세서·근태 근거로 급여 차이와 항목을 설명 | 급여 지급액 확정, 임의 공제 결정, 법 위반 판단 |
| `WORK_INSTRUCTION` | 근무시간·연락절차·작업행동을 근로자에게 안내 | 생산설비 제어, 자동 배치, 품질 예측 |
| `EMPLOYMENT_CHANGE` | 퇴사·무단결근·연락두절·이탈 등 사건의 확인과 후속업무 | 단순 계약 만료자 조회와 재계약 준비 |
| `OUT_OF_SCOPE` | 법률 최종판단, 기관 자동제출, 영상감시, 생산예측 등 지원범위 밖 요청 | 지원 가능한 요청에 정보만 부족한 경우 |

## 3. 자주 충돌하는 경계 사례

### 등록과 서류 요청

- `계약서 보고 새 직원 등록해줘` → `WORKER_ONBOARDING`
- `등록하려고 여권 사본을 직원에게 요청해줘` → `DOCUMENT_REQUEST`
- `직원 등록하고 여권 사본도 요청해줘` → 두 Intent 모두 선택

### 갱신과 고용변동

- `체류기간이 끝나는 사람 확인해줘` → `EXPIRY_RENEWAL`
- `계약을 연장하지 않는 직원의 후속 신고 준비해줘` → `EMPLOYMENT_CHANGE`
- `재계약 여부를 확인하고 종료자 신고도 준비해줘` → 두 Intent 모두 선택

### 급여 설명과 범위 밖 요청

- `지난달보다 월급이 적은 이유를 명세서로 설명해줘` → `PAYROLL_EXPLANATION`
- `이 직원 월급을 법적으로 얼마 줘야 하는지 결정해줘` → `OUT_OF_SCOPE`
- `회사 사정으로 다음 달에 지급한다고 안내해줘` → 요청 유형은 급여지만 위험한
  외부발송이므로 `PAYROLL_EXPLANATION`과 HR 검토 대상. 분류 자체를 범위 밖으로 만들지 않음

### 정보 부족과 범위 밖

- `금요일까지 그 서류 내라고 해줘` → `DOCUMENT_REQUEST`, 정보 부족
- `EPS에 로그인해서 신고를 완료해줘` → `OUT_OF_SCOPE`, 기관 자동제출 미지원

정보가 부족한 요청도 Intent를 알 수 있으면 해당 Intent를 부여합니다. `OUT_OF_SCOPE`를
모호성 라벨처럼 사용하지 않습니다.

## 4. Domain 결정표

| Domain | 대상 |
| --- | --- |
| `WORKER_PROFILE` | 이름·국적·언어·식별정보 등 근로자 기본정보 |
| `STAY` | 체류기간·체류자격·체류연장 |
| `CONTRACT` | 근로계약 기간·갱신·계약내용 |
| `DOCUMENT` | 여권 사본·외국인등록증 사본 등 제출 증빙 |
| `INSURANCE` | 4대보험·자격취득·보험 정보변경 |
| `PAYROLL` | 기본급·수당·공제·급여명세서 |
| `ATTENDANCE` | 출퇴근·결근·지각·근태 기록 |
| `WORK_SCHEDULE` | 근무시간·교대조·작업·일정 안내 |
| `EMPLOYMENT_STATUS` | 입사·퇴사·연락두절·고용변동 |
| `GENERAL_ADMIN` | 재직증명서 등 일반 사무·증명 업무 |

여러 Domain이 실제 요청 해결에 직접 필요하면 복수 선택합니다. 단순히 배경에 언급된
Domain은 선택하지 않습니다.

## 5. 복합 요청 판단

다음 두 조건을 모두 만족하면 복합 요청입니다.

1. 서로 다른 결과물 또는 업무카드가 필요함
2. 한 요청을 삭제해도 다른 요청을 독립적으로 수행할 수 있음

예시:

- `체류연장 준비하고 여권 사본도 요청해줘`
  → `EXPIRY_RENEWAL | DOCUMENT_REQUEST`
- `야간수당이 없는 이유를 확인해서 베트남어로 설명해줘`
  → 설명과 번역은 하나의 결과물이므로 `PAYROLL_EXPLANATION` 하나

복합 요청은 반려하지 않고 Intent별 후보 업무카드로 분리한 후 HR이 승인·삭제합니다.

## 6. 검수와 전문가 역할

- 팀 검수: Intent·Domain, 복합 여부, 서비스 범위 판정
- 전문가 검수: 고위험 Workflow, 법정기한, 필요서류, 기관절차, 조치안
- 모델팀 학습 대상: 팀 합의가 완료된 Intent·Domain만 우선 사용
- Agent 평가 대상: 전문가 확인이 끝난 Workflow·Slot·다음 행동만 사용

팀원이 실제 업무를 몰라도 문장의 요청 목적은 분류할 수 있습니다. 반대로 법령과 신고
절차가 맞는지는 팀 합의만으로 확정하지 않습니다.
